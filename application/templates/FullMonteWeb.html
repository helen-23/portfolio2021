    Introduction

    Background

    Interstitial photodynamic therapy (iPDT) is a modern cancer treatment method that eradicates tumors using light sensitive drugs activated by a set of light probes. In order to only activate drugs in the targeted tumor while minimizing damage to the healthy organ tissues surrounding it, the positions of the light probes and their intensities are the key.



    FullMonte and PDT-SPACE are software packages developed by the medical biophotonics research group and the computer engineering group of the University of Toronto, and together they serve to create optimized treatment plans for iPDT. PDT-SPACE intelligently searches for possible light probe setups, it then uses FullMonte to evaluate the various promising choices by simulating their light propagation and absorption behaviors, and it eventually outputs a set of information, including the optimized number of light probes needed, their powers, and their positions for the specified iPDT treatment plan.

    For more information on the FullMonte Project, please visit https://fullmonte.gitlab.io.

    Problem

    Despite being promising tools for iPDT treatment planning, both the FullMonte and PDT-SPACE software packages rely heavily on the command line interface and programming knowledge for setup and operations, making them impractical for medical practitioners to use. Therefore, there is a gap to build an intuitive and accessible user interface for PDT-SPACE and FullMonte in order to promote a more widespread adoption of the tools.

    Our Approach

    Our goal was to design a web interface that allows users who do not have extensive computer engineering knowledge to run FullMonte and PDT-SPACE, visualize the simulation and optimization results, and download output files. The idea was based on the previous work by Julian Gonsalves, Charlie Chai, Yuchao Jin, and Galen Brook-Atkins. Due to extention of the website's capabilities, most of the original work had to be re-written and many new features were added, such as supporting AWS computing backend.

    Research

    Other Monte-Carlo simulator applications

    We looked into all the existing Monte-Carlo simulator applications and noticed the following:

    1. Most of these software are not intended for turbid (highly-scattering) media such as living tissue. An example application is GoldSim, which is specialized for general science and business. Therefore, they are not suitable.
    2. Many popular Monte Carlo (MC) light scattering codes do not come with a GUI, such as Monte Carlo Multi-Layered (MCML).
    3. For the applications that do have a GUI, they have some scaling limitations. For example, Multi-Scattering is a web-based simulator for light propagation in turbid media and it has a beautiful user interface, but it uses 3 backend servers owned by Lund University, so only three simultaneous simulations are possible.
    4. FullMonte is by far the fastest open-source tetrahedral MC light propagation software, so building an intuitive graphics user interface is highly beneficial.
    5. None of the existing desktop/web applications provide iPDT treatment planning features. Today, most clinicians manually created plans based on heuristics and previous attempts of iPDT, so the creation of PDT-SPACE to automate the process of iPDT planning is highly benefcial. Therefore, it would be great if the PDT-SPACE software package can be incorporated into the application as well.

    Web-based vs Desktop Application

    We learned that FullMonte simulations can be compute-intensive, expecially with the visualization component, which demands powerful GPU support. Furthermore, we would like the website to be as easily accessible as possible for all medical researchers in the field, so there should be minimal setup and learning from the user's end. A desktop application requires manual setup on the user's device and would impose a strict demand on the device's CPU Specifications, whereas a web-based application does not require any setup on the user's device and can leverage the use of cloud computing services. Therefore, we have determined that a web-based application is the better choice for us.

    Website Framework

    We have chosen to build the FullMonteWeb application on the Django web framework, which is an open source Python-based framework designed for developing web applications. This framework is very well designed for developing secure and scalable web applications while also providing functionalities readily for use, such as the user authentication system.

    Website Hosting Options

    We investegated into all the suitable options for hosting dynamic websites, including Heroku, Amazon Elastic Beanstalk, and Amazon Lightsail. Heroku was eventually chosen for its wonderful compatibility with Django and cleanest deployment process, but we also found that it is not very suitable for larger scale applications, so although it suits our current requirements, it is possible to migrate the website to Amazon’s web hosting services in the future.

    Cloud Computing Services

    Amazon Web Services (AWS) offers an extensive suite of cloud computing platforms and APIs, ranging from vitrual computers to database and storage. It is also easily scalable and provides great customer support. Our users simply need to get an AWS EC2 instance and run all the simulations on that virtual machine through our website.

    Design

    System-level Overview

    At a high level, the FullMonte web interface first generates scripts using optical properties, lightsource locations, and dose thresholds from user inputs. It then sends the generated scripts and the user-uploaded meshes to an AWS computing node, and executes the simulation or optimization software. When the execution is finished, the web interface saves the output files from the AWS computing node for visualization and makes it available for users to download. From the user’s perspective, the web interface abstracts away the underlying software and convoluted interconnections, and they only need to interact with buttons and text fields.

    Features

    1. User Authentication
    Users are required to register an account and login before accessing any website functionalities. This is to ensure that users will be able to come back and access their results later, and to ensure that nobody else has access to results that are not their own.

    2. Tutorial
    We have created this page to demonstrate all the how-to's for using the website, including how to set up an AWS EC2 cloud computing instance, how to launch simulations, etc.. There is a combination of writings, images, and videos for maximal clearity.

    3. AWS Authentication
    After the user creates their own EC2 instance, they need to allow our website to securely access the instance by providing its DNS name and .pem file through our AWS setup page. Next, our website will attempt to perform a remote SSH (Secure Shell) operation to connect to the instance. Once connected, it will detect whether the computing node is set up with the required environment and software (Docker, FullMonte, Paraview Visualizer, PDT-SPACE, and NVIDIA GPU driver if the server contains a GPU), then automatically set them up if not detected.

    4. FullMonte Simulation
    The following is a series of pages that guide the user to providing the files and parameters for a FullMonte simulation. As the first step, it asks users to upload mesh files, or to choose one from the list of mesh files that they used in previous simulations. Other input fields include unit of the mesh file (cm or mm), the types of energies to record, total energy used, optical material properties, locations of the light sources, etc.. In the backend, our website takes all the selections and inputs and generates a TCL script with the appropriate functions for running FullMonte.

    Users can save material presets for a quick selection next time. Advanced users who knows how to work with TCl scripts can download the generated TCL script so far, complete the rest of the script manually, upload the completed sctipt, and run the simulation directly to skip the rest of the setup procedures.

    Users can visualize the input mesh in 3D to select a preferred light source location.

    The confirmation page allows users to check and many any final changes to the TCL script before launching the simulation.

    Once confirmed, users will be redirected to this page while simulation is running. This will inform users of the simulation progress.

    5. PDT-SPACE Treatment Planning
    The following is a series of pages that guide the user to prividing the files and parameters for a PDT-SPACE optimization planning. As the first step, it asks users to select a mesh file, and specify other fields such as source type, placement type, light placement file, etc.. In the backend, our website takes all the selections and inputs and generates an OP script with the appropriate code for running PDT-SCRIPT.

    6. Interactive Visualizers

    This page can be launched up after a successful Fullmonte simulation or a successsful PDT-SPACE optimization (to visualize the optimized light source configuration). There are two visualization features: a 3D visualizer for interacting with the output 3D models and a Dose-Volume Histogram. The main visualization page is composed of a link to a third-party 3D visualization tool and an interactive Dose-Volume Histogram.

    A dose-volume histogram serves to show the distribution of fluence (energy intensity) throughout each body tissue region described in the simulation's output .vtk file. Users can interact with the plot by zooming in and out, toggling region data on and off, and choosing whether or not to show markers for data points. When markers are turned on, the user can hover the mouse over them to view their x and y values.

    To achieve the 3D visualization functionality, the team researched and attempted several possible solutions. The final design that best aligns with our requirements makes use of a third party 3D visualization software called ParaView Visualizer, and how our website interfaces with this software is shown in the conceptual diagram below.

    7. Simulation History
    All generated scripts, input files, and output files from FullMonte are saved safely to an Amazon S3 central storage system. This page allows users to download these files from storage when needed.

    Verification and Iterations

    Testing and verifications were conducted and evaluated against a set of functional requirements, constraints, and objectives. There is a mix of quantitative and qualitative measurements.

    Quantitative Measurements

    For example, one of our objectives was for the website to be responsive, and our goal for this objective was to have a redirect latency of less than 800 ms. To measure the redirect time for each web page, we use the performance analyzer in the developer tool of Google Chrome browser. We went through workflows for simulation and optimization three times, and then calculated the average redirect time for each webpage. If the average redirect time is less than 800 ms for all web pages, our design meets this objective.

    Qualitative Measurements

    For example, one of our functional requirements was that the website should be intuitive. With the help of Dr. Lothar Lilge, a Ph.D. student in the medical biophysics group, and a few high school students, we were able to test out the flow of our website with real users. We received many helpful feedback on how to improve the website's usability and re-iterated through the project development phase to improve the website.
